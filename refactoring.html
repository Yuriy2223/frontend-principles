КРОК 0 З 4
Вхідний код
🚨 God Component

Ось наш «пацієнт» — UserCard. Один компонент робить все: завантажує дані, відображає UI, перевіряє права доступу, форматує вивід і логує аналітику.

Порахуємо порушення разом.

Проблеми
🔴
fetch прямо в компоненті — DIP
🔴
отримує весь User, використовує 2 поля — ISP
🔴
if/else по ролях захардкоджено — OCP
🔴
4 причини змінитись — SRP
КРОК 1 З 4
Single Responsibility
S — SRP

Виносимо логіку завантаження даних у окремий хук useUser. Компонент більше не знає про fetch — він тільки отримує готові дані і рендерить.

Одна причина для зміни замість чотирьох.

Зміни
🟢
Створено хук useUser(id) — вся логіка даних тут
🟢
Компонент тепер просто викликає хук
🔴
Прибрано useState/useEffect з компонента
КРОК 2 З 4
Dependency Inversion
D — DIP

Хук useUser досі залежить від fetch напряму. Передаємо HTTP-клієнт як параметр — тепер хук не знає про конкретний транспорт.

Замінити fetch на axios або React Query — нуль змін у бізнес-логіці.

Зміни
🟢
httpClient передається як залежність
🟢
Хук залежить від абстракції, не від fetch
🔴
Прибрано прямий виклик fetch
КРОК 3 З 4
Interface Segregation
I — ISP

Компонент отримував весь об'єкт User з 10+ полями, але використовував лише name і avatar. Звужуємо інтерфейс до мінімуму.

Тепер компонент незалежний від решти User.

Зміни
🟢
Пропс змінено з User на UserCardProps
🟢
Інтерфейс містить тільки потрібні поля
🔴
Прибрано надлишкові поля з пропсів
КРОК 4 З 4
Open / Closed
O — OCP

Логіка ролей була захардкоджена як if/else. Замінюємо на об'єкт-стратегію — тепер нова роль додається одним рядком без зміни самої функції.

Відкритий для розширення. Закритий для модифікації.

Зміни
🟢
Створено ROLE_LABELS — об'єкт конфігурації
🟢
Нова роль = один рядок у конфізі
🔴
Видалено if/else chain
ФІНАЛ
Рефакторинг завершено
✓ Чистий код

З одного компонента на 42 рядки ми отримали чисту архітектуру з розподіленими обов'язками. Кожна одиниця робить одну річ і легко тестується.
