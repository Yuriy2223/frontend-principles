SOLID принципи

S
Single Responsibility Principle
Принцип єдиної відповідальності

У модуля має бути лише одна причина для зміни. Висока зв'язність (Cohesion) всередині — добре. Високе зачеплення (Coupling) між модулями — погано.

❌ ПОГАНО — "Божественний хук"
const useUserActions = () => { const [user, setUser] = useState(null); // Причина 1: правила валідації const validate = (data) => data.email.includes('@'); // Причина 2: зміна API (REST→GraphQL) const save = async (data) => { if (validate(data)) { await fetch('/api/user', { body: JSON.stringify(data) }); } }; return { user, save }; };
✓ ДОБРЕ — Розділяємо відповідальність
// Хук 1: тільки логіка форми const useUserForm = (onSave) => { const [data, setData] = useState({}); const validate = () => { /*...*/ }; const handleSubmit = () => { if(validate()) onSave(data); }; return { data, handleSubmit }; }; // Хук 2: тільки транспорт const useUserApi = () => { const saveUser = async (payload) => { /* тільки fetch */ }; return { saveUser }; };
O
Open/Closed Principle
Принцип відкритості/закритості

Сутності мають бути відкриті для розширення, але закриті для модифікації. Нова фіча — без торкання старого робочого коду.

❌ ПОГАНО — Редагуємо щоразу
// При кожній новій ролі — лізем сюди const checkAccess = (user) => { if (user.role === 'admin') return true; if (user.role === 'editor') return user.hasPermission; if (user.role === 'guest') return false; };
✓ ДОБРЕ — Стратегія через конфіг
// Нова роль = просто нове поле const ROLES_STRATEGY = { admin: () => true, editor: (user) => user.hasPermission, guest: () => false, // + superadmin: () => true }; const checkAccess = (user) => { return ROLES_STRATEGY[user.role] ?.(user) ?? false; }; // checkAccess — не змінюється ніколи!
L
Liskov Substitution Principle
Принцип підстановки Лісков

Компоненти-обгортки не повинні ламати очікувану поведінку базових компонентів. Підклас завжди замінний базовим класом.

❌ ПОГАНО — Контракт порушено
interface ButtonProps { onClick: () => void; label: string; } // Забули onClick або змінили сенс const DeleteButton = ( { label }: { label: string } ) => { return ( <button onClick={() => confirm('Впевнені?')} > {label} </button> ); };
✓ ДОБРЕ — Контракт збережено
// Зберігаємо повний контракт const IconButton = ({ icon, ...props }: ButtonProps & { icon: string }) => { return ( <button {...props}> <Icon name={icon} /> {props.label} </button> ); }; // onClick прокидається через ...props
I
Interface Segregation Principle
Принцип розділення інтерфейсу

Клієнти не повинні залежати від інтерфейсів, які не використовують. Краще багато маленьких інтерфейсів, ніж один "товстий".

❌ ПОГАНО — Зайва залежність
// Avatar отримує весь User об'єкт, // хоча потрібен лише URL const Avatar = ( { user }: { user: User } ) => { return ( <img src={user.settings.profile.avatarUrl} /> ); }; // Тепер Avatar знає про User і залежить від нього!
✓ ДОБРЕ — Мінімальна залежність
// Залежимо лише від потрібного const Avatar = ( { src }: { src: string } ) => { return <img src={src} />; }; // Виклик: <Avatar src={user.settings.profile.avatarUrl} /> // Avatar не знає про User — він вільний!
D
Dependency Inversion Principle
Принцип інверсії залежностей

Модулі верхнього рівня не залежать від нижнього рівня. Не імпортуйте axios або localStorage прямо в бізнес-логіку — передавайте як залежність.

❌ ПОГАНО — Жорстка залежність
import axios from 'axios'; // Бізнес-логіка знає про axios! const getBonusPoints = (price: number) => { // Жорстко прив'язані до axios const rate = 0.1; return price * rate; }; // Замінити axios на fetch = переписати все
✓ ДОБРЕ — Залежність через абстракцію
// Бізнес-логіка не знає про транспорт const getBonusPoints = ( price: number, calculator: (v: number) => number ) => { return calculator(price); }; // Реалізація — окремо const simpleCalc = (p) => p * 0.1; // Виклик: getBonusPoints(100, simpleCalc);
Тріада здорового глузду

DRY · KISS · YAGNI

DRY
Don't Repeat Yourself

Це не про копіювання тексту — це про знання. Однакова логіка в двох місцях — порушення. Але два компоненти з однаковим кольором — ще не підстава для абстракції.

Пам'ятайте: «Неправильна абстракція» гірша за дублювання. Занадто рання DRY-оптимізація створює монстрів.

KISS
Keep It Simple, Stupid

Найкращий код — зрозуміє Junior. Якщо задачу можна вирішити через if/else — не треба будувати машину станів або useReducer.

Складність має бути виправданою. Питайте себе: чи ускладнення вирішує реальну проблему?

YAGNI
You Ain't Gonna Need It

Не пишіть код "на виріст". «А раптом знадобиться підтримка китайської?» — Коли знадобиться, тоді й додасте.

Зараз це просто зайве «сміття», яке треба тестувати, підтримувати і яке може зламатися.

Спершу зробіть так, щоб працювало. Потім зробіть так, щоб було красиво (рефакторинг за SOLID). Завжди пам'ятайте про компроміси: SOLID додає файлів, але зменшує кількість багів у майбутньому.

Практика

Тест на «вошивість»
вашого коду

1
Тест SRP
Чи можу я описати, що робить цей хук одним реченням — без сполучника «та»?
Якщо ні — діліть на менші частини.
2
Тест KISS
Чи зрозуміє цей код стажер через 15 хвилин без моїх пояснень?
Якщо ні — спрощуйте або документуйте.
3
Тест YAGNI
Чи використовую я цей generic тип або enum прямо зараз, чи це «про всяк випадок»?
Якщо «про всяк» — видаляйте сміливо.
4
Тест DIP
Якщо завтра ми замінимо Axios на React Query, скільки файлів бізнес-логіки треба переписати?
В ідеалі — нуль.

Когнітивне
навантаження

Головна мета всіх цих зусиль — зменшити когнітивне навантаження. Програмування — це керування складністю.

😵 Поганий код
10 змінних одночасно в голові
5 станів, 3 API-запити в одному файлі
«Бог-хуки» що роблять все і нічого
Помилку шукаєш 5 годин
Страх змінювати — «як би не зламати»
😌 Хороший код (за SOLID)
Маленька функція, одна річ
Мозок відпочиває при читанні
Кожен модуль — незалежна одиниця
Помилку знаходиш за 5 хвилин
Нова фіча — без страху зламати

Золота порада

Крок 1
Зробіть так, щоб працювало

Не думайте про архітектуру на самому початку. Ваша ціль — робочий продукт.

Крок 2
Зробіть так, щоб було красиво

Рефакторинг за SOLID. Коли код працює — час зробити його підтримуваним.

Крок 3
Пам'ятайте про компроміси

SOLID додає файлів. Але зменшує кількість багів у майбутньому. Це не ціна — це інвестиція.
