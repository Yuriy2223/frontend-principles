Всі принципи
на одній сторінці

Швидкий довідник для Code Review. Кожен принцип — одне речення, одне правило, один тест.

—
Принцип Запитай себе Якщо «так» — що робити
SRP
Чи можу описати це одним реченням без «та»?
Ні → ділити на менші одиниці
OCP
Чи треба лізти в старий код для нової фічі?
Так → виноси стратегію в конфіг/словник
LSP
Чи може нащадок замінити батька без змін у споживачі?
Ні → зберігай контракт через ...props

Чи використовує компонент всі отримані пропси/поля?
Ні → передай лише те що потрібно
DIP
Якщо замінити fetch/axios — скільки файлів бізнес-логіки зміниться?
> 0 → передавай транспорт як залежність
DRY
Чи є ця логіка ще десь у коді?
Так → виноси в утиліту/хук
KISS
Чи зрозуміє це джуніор за 15 хвилин?
Ні → спрощуй або документуй
YAGNI
Чи використовується цей код прямо зараз?
Ні → видаляй. Git зберігає все
SOLID
5 принципів
S
SRP
Single Responsibility Principle
Один модуль — одна причина для зміни.
Висока зв'язність (Cohesion) всередині модуля — добре. Високе зачеплення (Coupling) між модулями — погано.
→ Тест: Чи можеш описати що робить цей хук без сполучника «та»?
Детальніше в гайді →
O
OCP
Open/Closed Principle
Відкритий для розширення. Закритий для модифікації.
Нова фіча = новий рядок у конфізі, не нова умова в існуючій функції. if/else chain — майже завжди порушення OCP.
→ Тест: Чи треба змінювати цей файл для кожної нової ролі/типу/каналу?
Детальніше в гайді →
L
LSP
Liskov Substitution Principle
Підклас замінний батьківським без змін у споживачі.
Якщо компонент-нащадок змінює контракт (прибирає onClick, додає обов'язкові нові пропси) — це LSP. Рятує ...props.
→ Тест: Чи може споживач замінити Button на IconButton без змін у своєму коді?
Детальніше в гайді →
I
ISP
Interface Segregation Principle
Краще багато маленьких інтерфейсів, ніж один товстий.
Компонент не повинен залежати від полів які він не використовує. Замість user: User передай лише src: string.
→ Тест: Чи використовує компонент всі поля отриманого об'єкта?
Детальніше в гайді →
D
DIP
Dependency Inversion Principle
Залежи від абстракцій, а не від конкретних реалізацій.
Не імпортуй axios або localStorage прямо в бізнес-логіку. Передавай http-клієнт як параметр — заміна без змін у логіці.
→ Тест: Якщо замінити fetch на axios — скільки файлів бізнес-логіки зміниться?
Детальніше в гайді →
Тріада здорового глузду
DRY · KISS · YAGNI
DRY
Don't Repeat
Don't Repeat Yourself
Кожне знання має єдине представлення в системі.
Це не про схожий текст — це про знання. Однакова логіка ПДВ у трьох файлах = порушення. Два компоненти з однаковим кольором = не обов'язково.
→ Тест: Якщо зміниться ставка ПДВ — скільки файлів треба оновити?
Детальніше в гайді →
KISS
Keep Simple
Keep It Simple, Stupid
Найкращий код — той що зрозуміє джуніор.
Якщо задачу можна вирішити через if/else — не треба useReducer і машини станів. Складність має бути виправданою реальною потребою.
→ Тест: Чи зрозуміє новий член команди цей код за 15 хвилин без пояснень?
Детальніше в гайді →
YAGNI
Not Needed
You Ain't Gonna Need It
Не пиши код який «може знадобитись» в майбутньому.
«А раптом знадобиться» — найдорожча фраза в розробці. Git зберігає все — коли знадобиться, відновиш за 10 секунд. Зараз це технічний борг.
→ Тест: Чи використовується цей generic/enum/параметр прямо зараз?
Детальніше в гайді →
Антипатерни
6 помилок
God Component
Порушує: SRP, OCP
Компонент що знає все і робить все.
Симптоми: файл 500+ рядків, 10+ useState, useEffect з 5 залежностями. Рішення: виділити хуки і дочірні компоненти.
→ Тест: Чи можеш написати unit-тест за 5 хвилин?
Приклад коду →
Prop Drilling
Порушує: ISP, SRP
Дані пробрасовуються через 3+ рівні де їх не використовують.
Рішення: Context API, Zustand, або React Query для серверного стану. Глобальне — глобально.
→ Тест: Чи проходить проп більше 2 рівнів не використовуючись?
Приклад коду →
Magic Numbers
Порушує: DRY, KISS
Числа і рядки без назви прямо в логіці.
86400000 нічого не каже. SESSION_TTL_MS — каже все. Всі константи в constants.ts.
→ Тест: Чи знаєш ти що означає кожне число в цьому файлі без контексту?
Приклад коду →
Shotgun Surgery
Порушує: OCP, DRY
Одна зміна вимагає правок у десятках файлів.
Якщо PR торкається 10+ файлів у різних модулях — знайди спільну абстракцію і виноси в util/helper.
→ Тест: Якщо зміниться формат імені юзера — скільки файлів треба оновити?
Приклад коду →
Dead Code
Порушує: YAGNI, KISS
Код який ніколи не виконується але займає місце.
Закоментовані блоки, unused imports, застарілі feature flags. Git зберігає все — видаляй без жалю.
→ Тест: Коли останній раз виконувався цей закоментований блок?
Приклад коду →
Premature Opt.
Порушує: YAGNI, KISS
Оптимізація до того як виміряно проблему.
useMemo/useCallback без реального виміру — overhead. Правило Кнута: «Передчасна оптимізація — корінь усього зла».
→ Тест: Чи є дані з React DevTools Profiler що виправдовують це useMemo?

Мета-поняття
Терміни
Cohesion
Зв'язність
Наскільки елементи всередині модуля логічно пов'язані.
Висока зв'язність = добре. Всі методи хука стосуються однієї теми. Низька зв'язність = God Component.
Coupling
Зачеплення
Наскільки модулі залежать один від одного.
Низьке зачеплення = добре. Компонент Avatar не знає нічого про User крім src. Високе зачеплення = крихкий код.
Abstraction
Абстракція
Інтерфейс без деталей реалізації.
HttpClient як тип замість конкретного fetch — абстракція. Дозволяє замінювати реалізацію без зміни споживача.
Wrong Abstraction
Неправильна абстракція — Сенді Метц
Занадто рання DRY-оптимізація створює монстрів.
Дублювання дешевше за неправильну абстракцію. Спочатку дай коду «дозріти», потім виноси спільне. Не поспішай.
Tech Debt
Технічний борг
Майбутня вартість сьогоднішніх компромісів.
Свідомий борг — ок (MVP). Несвідомий борг — небезпечно. SOLID зменшує борг, але збільшує кількість файлів.
Refactoring
Рефакторинг
Покращення структури коду без зміни поведінки.
Спершу зробіть так щоб працювало. Потім — щоб було красиво. Ніколи не рефактори і не додаєш фічу одночасно.
